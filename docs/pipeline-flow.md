# パイプライン運用フロー（v3.1 / 2026-07-23整理）

「ネタの収集 → 審査 → 広告テスト → 開発引き渡し」の全体像。各段の担当と、次の段に進む条件。

```
┌─ 自動化層（非LLM・GitHub Actions cron・毎日/毎週）─────────────┐
│ 収集bot4本 → bot/data/signals/*.jsonl に蓄積                     │
│   laws(週・施行3-4ヶ月前窓) / prtimes(日・サ終値上げ等の供給側)   │
│   gplay(日・監視アプリの★1-2) / youtube(日・ニーズ語コメント)     │
└──────────────┬───────────────────────────────┘
               ▼
┌─ セッション層・週次 /weekly-pipeline ──────────────────────┐
│ Step 0-1: 死活確認 → 未処理信号読み込み → explored-map照合        │
│ Step 2:   トリアージ（①スキャン: 無料/内蔵AIで既に解決済みか）    │
│ Step 3-8: 深掘り・競合・需要・ナレッジ照合                        │
│ Step 9:   ゲート1審査（市場4軸10点）                              │
│ Step 12:  GitHub Issue台帳へ（pass=[検証済] / hold=[保留] / reject=クローズ）│
└──────────────┬───────────────────────────────┘
               ▼ pass（スコア7+）のみ
┌─ 広告テスト /test-marketing ─────────────────────────────┐
│ LP・X投稿・note素材の生成 → 広告出稿（※出稿はユーザー承認制）     │
│ 2-4週間 / 日額¥500目安 / KPI: リード獲得数（Go基準は仮説ごと設定） │
│ 計測: 広告クリック→LP内エンゲージ→メール登録の多段ファネル        │
└──────────────┬───────────────────────────────┘
               ▼ Go判定のみ
┌─ 開発引き渡し（事業ノート リサーチ（AIプロダクト））────────────┐
│ ゲート2: 一番難しい部分はコードの中か外か                         │
│ ゲート3: ①存続性 + ③蓄積（2問両方。スキル・資産前提は使わない）   │
│ → 採用ネタ/ に1ネタ1ファイル → AI自律開発ハーネス フェーズ1へ     │
└──────────────────────────────────────────────┘
```

## 原則

- **候補台帳の正本はGitHub Issues**（事業ノート側にリストを作らない）
- **広告テストの前に人手の判断が入るのは「出稿」の1点だけ**（費用が発生するため）。素材生成までは自動で進めてよい
- **KPIはリード数**。pass率・スコアは内部指標であり成果と呼ばない（#39: score 8→リード0の教訓）
- 広告テストで死んだ仮説は explored-map に判定を書き戻し、同じ轍を踏まない
