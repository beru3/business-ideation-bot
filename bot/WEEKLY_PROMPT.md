# 週次パイプライン: 信号深掘り → 仮説生成 → 市場検証

> Claude Code で週1回実行。PC起動時に `claude` で本ファイルを指定する。

## 概要

Actions が収集した生信号を起点に、Claude Code が以下を一貫実行する:
1. 信号の読み込みとトリアージ
2. 有望信号の深掘りリサーチ
3. 仮説生成
4. 市場検証 + スコアリング
5. マーケブリーフ生成
6. Issue投稿・ラベル変更

## 実行手順

```bash
# Step 0: 最新の信号ファイルを確認
cat bot/signals_raw.json | python -c "import json,sys; d=json.load(sys.stdin); print(f'信号: {d[\"total_queries\"]}クエリ, {len(d.get(\"signals\",[]))}件の検索結果')"

# Step 1-4: Claude Code が対話的に実行（以下の指示に従う）

# Step 5: 結果をIssueに反映
python bot/validate.py post
```

---

## Step 1: 信号トリアージ

`bot/signals_raw.json` の検索結果を読み、以下の基準で有望な信号を選別する。

**選別基準:**
- 具体的なペインの表明がある（「困っている」「面倒」「高すぎる」等）
- 複数ソースから同じ課題が検出されている
- 既存ソリューションへの不満が具体的
- 法改正・制度変更など強制需要の兆候
- 海外で成功しているが日本未上陸のサービス

**除外基準:**
- 一般的すぎるトレンド記事（「DXが重要」等）
- 大手が既に支配している領域の信号
- ターゲットが曖昧な信号

出力: 有望信号リスト（10件以下に絞る）

---

## Step 2: 深掘りリサーチ

Step 1で選んだ有望信号それぞれについて、Web検索で深掘りする。

**深掘り観点:**
- この課題で困っている人は本当にいるか？（SNS投稿、フォーラム、レビュー）
- 既存の解決策は何か？なぜ不十分か？
- 市場規模のヒント（対象企業数、検索ボリューム）
- 類似サービスの価格帯と評判

**深掘りのコツ:**
- 1つの信号から「そもそもなぜこの問題が起きるのか？」を掘る
- 表面的な課題の裏にある構造的な原因を探す
- 「この課題を解決するのに月いくら払うか？」の手がかりを探す
- 検索結果が面白ければ、追加クエリを自分で生成して検索を続ける

---

## Step 3: 仮説生成

深掘りの結果、事業機会がありそうなものについて仮説を構造化する。

**仮説フォーマット:**
```
タイトル: [サービス名案]
ターゲット: 誰の
課題: どんな痛みを
ソリューション: どう解決するか
価格帯: いくらで
差別化: なぜ既存ではダメで、これなら勝てるか
信号の根拠: どの信号からこの仮説に至ったか
```

**品質基準:**
- 「ターゲット」が具体的（「中小企業」ではなく「従業員20-100名のEC事業者の経理担当」）
- 「課題」がターゲットの言葉で表現されている
- 「差別化」に競合名と具体的な隙が書かれている
- 価格帯の根拠がある（競合価格、ターゲットの支払い能力）

---

## Step 4: 市場検証 + スコアリング

### 4-1. ナレッジ照合

`bot/validate_input.json` のナレッジインサイト（496件）から関連知見を検索する。
（validate_input.jsonがなければ `python bot/validate.py fetch` で生成）

確認観点:
- 類似ターゲットの既存ペイン情報
- 類似ソリューションの成功/失敗パターン
- マーケティング手法の有効性データ

### 4-2. 競合調査

Web検索で以下を調べる:
- 同じ課題を解決する既存サービス（日本語・英語）
- 価格帯、機能の網羅度、ターゲット
- 弱点・隙

### 4-3. 需要裏付け

- 関連キーワードの検索ボリューム推定
- SNS/フォーラムでの類似悩み投稿の量
- 業界レポートや統計

### 4-4. スコアリング

| 軸 | 基準 | 点数 |
|---|---|---|
| 競合の隙 | 直接競合なし=3 / 弱い競合のみ=2 / 差別化可能=1 / 大手独占=0 | 0-3 |
| 需要確認 | 明確な証拠=3 / ある程度=2 / 微妙=1 / なし=0 | 0-3 |
| 収益成立性 | 明確に成立=2 / ギリギリ=1 / 非現実的=0 | 0-2 |
| タイミング | 今がベスト=2 / 悪くない=1 / 遅い/早すぎ=0 | 0-2 |

**判定:**
- 7点以上 → `pass`（検証済、テストマーケへ）
- 4-6点 → `hold`（保留）
- 3点以下 → `reject`（却下）

---

## Step 5: マーケブリーフ生成（passのみ）

Phase 3でLP・X投稿・note記事を生成するための指示書を作る。

```json
{
  "target_persona": "誰に売るか（具体的に）",
  "pain_statement": "ターゲットが共感する痛みの表現",
  "solution_hook": "一文で伝わるソリューション説明",
  "differentiator": "競合との違い（なぜこれを選ぶか）",
  "cta_design": "CTAの設計（何をしてもらうか）",
  "pricing_hint": "想定価格帯",
  "channels": ["有効なチャネル"],
  "lp_angle": "LPの訴求角度",
  "x_hooks": ["X投稿で使えるフック3つ"],
  "knowledge_backed_tactics": ["ナレッジから得たマーケ戦術"]
}
```

---

## Step 6: 出力

全結果を `bot/validate_output.json` に書き込む。

```json
{
  "validated_at": "ISO8601",
  "results": [
    {
      "issue_number": null,
      "issue_title": "新規仮説のタイトル",
      "verdict": "pass|hold|reject",
      "market_fit_score": 8,
      "score_competitive_gap": 2,
      "score_demand_evidence": 3,
      "score_revenue_viability": 2,
      "score_timing": 1,
      "competitive_analysis": "競合調査テキスト",
      "demand_evidence": "需要裏付けテキスト",
      "knowledge_match": "ナレッジ照合テキスト",
      "marketing_brief": { ... },
      "source_signals": ["元になった信号のクエリ/URL"]
    }
  ]
}
```

issue_number が null の場合は新規Issue作成、既存番号ならコメント追加。

完了後:
```bash
python bot/validate.py post
```

---

## 注意事項

- 厳しめに評価すること。通過率30-50%が健全
- 「なんとなく良さそう」では pass にしない。具体的な証拠が必要
- ナレッジに「この手のツールは失敗する」知見があれば素直に reject
- 信号の深掘りは「これ以上掘っても新情報が出ない」と判断するまで続ける
- 1セッションで5-10仮説を処理する想定（15分程度）

## フィードバックループ

テストマーケ結果が返ってきたら、以下を次回の信号収集に反映:
- 失敗した仮説の領域は検索クエリから除外 or 重み下げ
- 成功した仮説の隣接領域を検索クエリに追加
- ナレッジDBに結果を蓄積（Supabase test_campaigns テーブル）
